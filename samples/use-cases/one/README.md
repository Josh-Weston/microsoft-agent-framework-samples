# Use Case 1: Sequential Orchestration for "Robotic Triage" and Compliance Checking

## Architectural Concept

Public administration involves vast quantities of routine document processing, ranging from business permit applications to complex regulatory filings. Agentic AI is optimally positioned to handle robotic triage work, autonomously parsing unstructured documents, extracting required metadata, and validating that metadata against rigid compliance protocols. By offloading sequential verification tasks, civil servants are liberated to address complex, high-stakes edge cases that require genuine human empathy, nuanced judgment, and discretionary authorit.

## Proof-of-Concept Scenario

This proof-of-concept simulates an autonomous business permit application pipeline. The system processes a queue of applications, extracting data, checking zoning constraints, and preparing a final recommendation for a human adjudicator. The objective is to demonstrate that an agentic workflow can sequentially refine raw data into actionable intelligence without human intervention, up until the point of a binding administrative decision.

## Key Framework Patterns Explored

The architecture relies entirely on the Sequential Orchestration Pattern, initialized via the framework's SequentialBuilder. In a sequential workflow, agents are organized into a strict deterministic pipeline. Each specialized agent processes the task in turn, passing its analytical output to the next agent down the sequence.Within this specific scenario, developers must explore the chain_only_agent_responses configuration parameter. By default, each agent in a sequential chain consumes the entire conversation history of the preceding agents. However, for a strict compliance pipeline, it is often safer to configure the system so that downstream agents only receive the finalized, synthesized output of the previous step. This prevents the Compliance Agent from being biased by the raw, unstructured input data or any hallucinations generated during the initial extraction phase, forcing it to rely solely on the structured JSON produced by the Data Extraction Agent. Furthermore, the framework allows for the mixing of custom executors with LLM-backed agents. A custom executor (a code-only block that does not invoke an LLM) can be placed at the end of the sequence to format the final output, provided it adheres to the terminator contract by yielding an AgentResponse.

## Local Implementation Strategy

To construct this locally without cloud infrastructure, the developer initializes a local filesystem directory populated with dummy text files representing permit applications. Some files are perfectly completed, while others are deliberately missing critical information such as parcel numbers or applicant signatures. Two distinct agents are instantiated, powered by a local Ollama model (e.g., llama3.2) via the OllamaChatClient :The Data Extraction Agent: Programmed with precise system instructions to read the raw application text and output a structured JSON format containing entity names, addresses, and proposed business activities.The Compliance Agent: Programmed to ingest the structured JSON output and validate the extracted entities against a hardcoded set of municipal zoning rules and licensing prerequisites provided in its system prompt.The pipeline is executed locally by passing the agents into the SequentialBuilder in the required order: SequentialBuilder(participants=[data_agent, compliance_agent]).build(). The developer then writes a local loop that reads the dummy directory, passing each file's contents to the workflow execution engine.

## Human-in-the-Loop Mechanics

Safety and accountability are enforced through the framework's workflow pausing capabilities. The developer configures the workflow to utilize a request-info mechanism targeted specifically at the Compliance Agent by appending .with_request_info(agents=["compliance_agent"]) to the builder.If the Compliance Agent detects a zoning violation, or if the confidence score of the extracted data falls below a predefined threshold, the agent does not autonomously reject the permit. Instead, it yields a request event detailing its findings. In the developer's local terminal, the execution stream halts. The system prints a summary of the compliance failure and prompts the human operator. The developer can type an approval of the rejection, demand further automated review, or provide specific override instructions. Only upon receiving this human terminal input does the execution graph resume, utilizing the human's response to finalize the permit status.

## Second and Third-Order Implications

The sequential pattern ensures a highly auditable chain of custody. Because the flow of data is strictly linear, system administrators can isolate exactly which agent failed if an error occurs, mitigating the opaque failure modes of non-linear systems. Furthermore, by enforcing human intervention on rejections, the government mitigates the risk of algorithmic bias leading to adverse administrative actions against citizens without due process. The strict ordering guarantees that compliance checks are never bypassed, establishing a verifiable pipeline suitable for rigorous administrative law scrutiny.
